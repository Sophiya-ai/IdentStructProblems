# Проект identProblem – Идентификация структуры проблемы, а также работа с базой данных проблем и их взаимосвязей (PostgreSQL 17)

Проект на Python работает с базой данных `identProblem`, содержащей результаты работы 
метода идентификации структуры анализируемой проблемы, а именно: информацию о проблемах, классах и 
типах отношений между ними. Данные о проблемах хранятся в таблицах `subproblems` 
(включая JSONB-поля для макро- и микроуровневых моделей), а связи описываются через таблицы `relClass`, `relName` и `problem_relationships`. 
Проект предоставляет удобные CRUD-модули для каждой таблицы, а также поиск проблем по наименованию 
(с использованием JSONB-операторов PostgreSQL). Также предусмотрено создание и работа с локальной векторной БД (ChromaDB), 
если пользователь предоставляет файлы рекомендаций, методических указаний, инструкций и т.д., 
на которые он опирается в своей работе, в формате `.txt` и/или `.pdf`. Метод реализован посредством технологии БЯМ.

##  📌 Возможности

- Подключение к PostgreSQL через пул соединений (psycopg2)
- Полный набор CRUD-операций для таблиц:
  - `subproblems` – проблемы с поддержкой иерархии (parent_id)
  - `relClass` – классы отношений (таксономические, временные, причинно-следственные и т.д.)
  - `relName` – типы отношений ("is-a", "часть-целое", "приводит к" и др.)
  - `problem_relationships` – связи между проблемами
- Поиск подпроблем по тексту в полях, например, макромодели (`macro_model->>'sbj'`, `macro_model->>'sit'`)
- Безопасные параметризованные запросы
- Каскадное удаление за счёт ограничений на уровне БД (ON DELETE CASCADE)
- Конфигурация через переменные окружения (.env)
- Векторный поиск по файлам (локальная база знаний с ChromaDB)
- Идентификация структуры подпроблем каждой проблемы декомпозиции, а также генерация их макро- и микромоделей 
- Верификация результатов работы БЯМ на каждом этапе работы метода. Предлагаемый метод сочетает:
  - множественную генерацию (адаптированный вариант self consistency) 
  - оценку сильной БЯМ (LLM-as-a-Judge)
  - оценку уверенности на основе перплексии (Perplexity-based Confidence) 
  - _при наличии заполненной векторной БД_ – поиск по релевантным документам (Retrieval-Augmented Verification)
  - строгую структурную верификацию, гарантирующую корректность JSON-схемы ответа
- Визуализация подпроблем с низкой уверенностью верификации из дерева интересующей корневой проблемы
- Визуализация поддерева интересующей проблемы - выводятся идентификаторы из макроуровневой модели проблемы/подпроблемы с техническими идентификаторами БД
- Хранение последнего запрошенного визуализированного поддерева в файле [hierarchy.json](hierarchy.json)
- Проверка новой рассматриваемой проблемы на наличие в БД - была ли она или схожая с ней уже ранее декомпозирована?

## ✨ Требования

- Python 3.10+
- PostgreSQL 17 (или совместимая версия)
- Установленная база данных `identProblem` с необходимыми таблицами (см. описание структуры)
- Библиотеки из `requirements.txt`

## 🔧 Установка и настройка

1. Клонируйте репозиторий:
   ```bash
   git clone <url-репозитория>
   cd identProblem_project
   ```
2. Создайте виртуальное окружение и активируйте его:
   ```bash
    python -m venv venv
    source venv/bin/activate   # Linux/macOS
    venv\Scripts\activate      # Windows
   ```
3. Установите зависимости:
   ```bash
    pip install -r requirements.txt
   ```
4. Настройте переменные окружения. Создайте файл `.env` и отредактируйте, указав параметры подключения к вашей базе данных:
   
    ```text
    DB_HOST=localhost
    DB_PORT=5432
    DB_NAME=identProblem
    DB_USER=postgres
    DB_PASSWORD=your_password
    OPENROUTER_API_KEY=your_OPENROUTER_API_KEY
    OPENROUTER_MODEL=your_OPENROUTER_MODEL
    DEFAULT_GENERATION_MODEL=your_GENERATION_MODEL
    DEFAULT_SAMPLING_MODEL=your_SAMPLING_MODEL
    DEFAULT_JUDGE_MODEL=your_JUDGE_MODEL
    RETRIEVAL_ENABLED=true
    ```
5. Для верификации результатов работы БЯМ мы используем в качестве одной из оценок, оценку на основе вычисления перплексии. Здесь есть два варианта:
- **Сложный.** Использовать модель **Llama‑3.2** с Hugging Face (или другую подобную ей) – это закрытая (gated) модель с большим контекстным окном, требующая аутентификации и подтверждения лицензии. Тогда необходимо сделать следующее:
  - зарегистрироваться на Hugging Face, создать на [token](https://huggingface.co/settings/tokens) Access Token с правами read
  - обновить huggingface_hub:
  ```bash
    pip install --upgrade huggingface_hub
   ```
  - авторизоваться в Hugging Face – выполните в терминале (виртуальное окружение активно):
  ```bash
    hf auth login
    ```
  - вас попросят ввести Access Token. После успешного входа появится сообщение `Login successful`
  - принять лицензию модели **Llama‑3.2** – перейдите на страницу модели [meta-llama/Llama-3.2-1B-Instruct](https://huggingface.co/meta-llama/Llama-3.2-1B-Instruct), заполните короткую форму и согласитесь с условиями
  - проверьте, что доступ открыт:
  ```bash
    hf download meta-llama/Llama-3.2-1B-Instruct --include "config.json"
    ```
- **Простой.** Если доступ не дали или нет возможности использовать первый подход, то можно применить открытую модель, например: **DistilGPT2** – не требует авторизации и легковесна. Для этого:
  - в файле [perplexity.py](perplexity.py) раскомментируйте строки:
    ```bash
      # DEFAULT_MODEL_NAME = "distilgpt2"
      # DEFAULT_MAX_LENGTH = 1024
    ```
  - и закомментируйте:
    ```bash
      DEFAULT_MODEL_NAME = "meta-llama/Llama-3.2-1B-Instruct"
      DEFAULT_MAX_LENGTH = 8192
    ```
  - если контекстного окна `distilgpt2` не хватает, то можно использовать, например, `radce/Llama-3.2-1B-ru-v2` – открытая и легковесная (1B параметров) версия Llama‑3.2, не требует авторизации, контекстное окно – до 128k токенов, дообученная на русском языке и поддерживает чат‑шаблон Llama 3 (скорее всего, наследует его от исходной инструктивной версии)

6. Убедитесь, что структура таблиц в вашей базе данных соответствует описанию (см. ниже).

## 💾 Описание таблиц базы данных
### Проект рассчитан на следующую структуру (должна быть создана предварительно):

### 1. `subproblems` – иерархическая таблица проблем

| Поле               | Тип      | Ограничения                                                       | Описание                                                            |
|--------------------|----------|-------------------------------------------------------------------|---------------------------------------------------------------------|
| `id`               | `SERIAL` | `PRIMARY KEY`                                                     | Уникальный идентификатор проблемы (автоинкремент)                   |
| `parent_id`        | `INT`    | `REFERENCES subproblems(id) ON DELETE CASCADE`, может быть `NULL` | Ссылка на родительскую проблему (`NULL` для корневого узла)         |
| `macro_model`      | `JSONB`  | `NOT NULL`                                                        | Макроуровневая модель проблемы (субъект, ситуация, оценка)          |
| `micro_model`      | `JSONB`  | –                                                                 | Микроуровневая модель проблемы (концептуальные модели, метод и др.) |
| `confidence_macro` | `TEXT`   | –                                                                 | Уверенность по макроуровневой модели проблемы (`NULL` или `low`)    |
| `reasoning_macro`  | `TEXT`   | –                                                                 | Пояснение к полученной макроуровневой модели                        |
| `confidence_micro` | `TEXT`   | –                                                                 | Уверенность по микроуровневой модели проблемы  (`NULL` или `low`)   |
| `reasoning_micro`  | `TEXT`   | –                                                                 | Пояснение к полученной микроуровневой модели                        |
| `reasoning_prbfld` | `TEXT`   | –                                                                 | Пояснение к полученной модели проблемного поля                      |

### 2. `relClass` – классы отношений

| Поле                  | Тип      | Ограничения       | Описание                                                            |
|-----------------------|----------|-------------------|---------------------------------------------------------------------|
| `id_relClass`         | `SERIAL` | `PRIMARY KEY`     | Уникальный идентификатор класса отношений                           |
| `relClassName`        | `TEXT`   | `UNIQUE NOT NULL` | Наименование класса (таксономические, причинно-следственные и т.п.) |
| `descriptionRelClass` | `TEXT`   | –                 | Дополнительное описание класса                                      |

### 3. `relName` – типы отношений

| Поле             | Тип      | Ограничения                                      | Описание                                                                 |
|------------------|----------|--------------------------------------------------|--------------------------------------------------------------------------|
| `id_relName`     | `SERIAL` | `PRIMARY KEY`                                    | Уникальный идентификатор типа отношения                                  |
| `name`           | `TEXT`   | `UNIQUE NOT NULL`                                | Название отношения («is‑a», «часть‑целое», «приводит к» и т.д.)          |
| `id_relClass`    | `INT`    | `NOT NULL REFERENCES relClass(id_relClass)`      | Внешний ключ к таблице классов отношений                                 |
| `descriptionRel` | `TEXT`   | –                                                | Дополнительное описание типа отношения                                   |

### 4. `problem_relationships` – связи между проблемами

| Поле              | Тип      | Ограничения                                                 | Описание                                                               |
|-------------------|----------|-------------------------------------------------------------|------------------------------------------------------------------------|
| `id`              | `SERIAL` | `PRIMARY KEY`                                               | Уникальный идентификатор связи                                         |
| `subject_id`      | `INT`    | `NOT NULL REFERENCES subproblems(id) ON DELETE CASCADE`     | Идентификатор проблемы-субъекта (откуда исходит связь)                 |
| `id_relationship` | `INT`    | `NOT NULL REFERENCES relName(id_relName) ON DELETE CASCADE` | Идентификатор типа отношения                                           |
| `object_id`       | `INT`    | `NOT NULL REFERENCES subproblems(id) ON DELETE CASCADE`     | Идентификатор проблемы-объекта (куда направлена связь)                 |
| `metadata`        | `JSONB`  | –                                                           | Дополнительные атрибуты связи (вес, дата и т.п.)                       |
| -                 | –        | `UNIQUE(subject_id, id_relationship, object_id)`            | Запрещает дублирование одинаковых троек «субъект – отношение – объект» |
